# WordSprint 技术规范与维护基线

本文档定义 `WordSprint` 的产品边界、技术约束、模块职责、数据库与 Redis 设计原则以及测试规范。它不记录开发进度，当前项目状态以 `README.md`、`HANDOFF.md`、`ITERATION_TODO.md` 为准。

---

## 1. 项目定位

- 项目名：`WordSprint`
- 类型：轻量单词卡 / 记忆训练平台
- 目标：用于 `Java 学习`，不是商业化大系统
- 架构：`Vue 3 + Spring Boot + Redis + MySQL`

本项目关注的是一个小而完整、可持续维护的学习系统，不扩展为微服务、多租户或复杂教育平台。

---

## 2. MVP 范围

项目的核心边界仍然是以下九项：

1. 用户注册登录
2. 单词卡 CRUD
3. 学习训练
4. 答题提交与判定
5. 错题本
6. 打卡
7. 积分与排行榜
8. 学习统计
9. Redis 在缓存、排行榜、随机题池、签到状态中的实际使用

以下能力默认不做，除非用户明确要求：

- 邮箱验证码
- 消息通知系统
- 复杂推荐算法
- 多租户设计
- 分布式事务
- 微服务拆分
- Elasticsearch
- MQ
- Docker 编排

## 4. 后端模块拆分

后端包结构保持以下约定：

```text
common/
config/
controller/
dto/
entity/
mapper/
redis/
security/
service/
service/impl/
vo/
```

职责约定：

- `controller` 只负责请求接收与结果返回
- `service` 负责业务逻辑
- `mapper` 负责数据访问
- 输入使用 `DTO`
- 输出使用 `VO`
- `entity` 不直接返回前端

---

## 5. 前端结构约定

前端目录保持以下约定：

```text
src/api/
src/components/
src/layout/
src/router/
src/stores/
src/styles/
src/utils/
src/views/
```

职责约定：

- 页面级组件放 `views`
- 可复用组件放 `components`
- 所有接口统一走 `api` 层
- 路由守卫统一处理登录态
- Token 管理统一封装

## 7. 数据库设计原则

- 表名、字段名统一使用 `snake_case`
- 主键统一为 `id`
- 时间字段统一使用 `created_at`、`updated_at`
- 删除优先逻辑删除
- 高频查询字段建立索引
- 状态字段使用明确枚举值

如修改数据库设计，必须同步更新：

- `DATABASE_DESIGN.md`
- `sql/init.sql`
- `API_LIST.md`

---

## 8. Redis 设计原则

Redis 在本项目中的职责限定为：

- 排行榜
- 热点缓存
- 随机抽题池
- 签到状态
- 幂等锁 / 防重复提交

要求：

- Key 使用统一前缀：`wordsprint:`
- 临时数据必须设置 TTL
- 核心数据不能只保存在 Redis
- 更新数据库后要考虑缓存失效或刷新

---

## 9. API 设计原则

- 基础前缀统一为 `/api`
- 风格以 REST 为主
- 查询优先 `GET`
- 新增优先 `POST`
- 更新优先 `PUT`
- 删除优先 `DELETE`
- 返回结构统一
- 涉及私有数据必须校验当前用户归属权

---

## 10. 安全约束

- 密码必须加密存储
- 不记录明文密码
- 不打印敏感 token
- 私有数据接口必须校验当前用户
- 不允许越权访问他人词卡、错题或学习记录

---

## 11. 测试规范

### 11.1 后端测试重点

- 用户注册登录
- JWT 鉴权有效性
- 单词卡 CRUD 权限校验
- 答题提交正确性
- 错题本状态变化
- 连续打卡计算
- Redis 排行榜同步逻辑

### 11.2 前端测试重点

- 登录态跳转
- 页面表单校验
- 学习流程交互正确性
- 错题本和排行榜显示正确
- 异常状态提示友好

### 11.3 最小手工验证

- 新用户能否完整注册登录
- 单词卡能否新增、编辑、删除、查询
- 学习答题后是否写入学习记录
- 答错后是否进入错题本
- 打卡是否只在当天首次学习后建立
- 排行榜是否随积分变化而变化

---

## 12. 性能与维护建议

- 热门统计和排行榜优先走 Redis
- 列表查询必须分页
- 高频查询字段建立索引
- 学习提交接口保持防重复机制
- 大批量导入词卡时使用批处理
- 图表统计接口避免返回过大的原始数据集

---

## 13. 上线前检查项

- 配置文件中无明文敏感信息提交到仓库
- 默认账号和测试数据已处理
- MySQL 和 Redis 连接配置正确
- 跨域配置可用
- 生产环境关闭调试日志
- README 补齐启动说明
- 核心接口完成自测

---

## 14. 当前维护顺序建议

如果后续继续开发，建议按这个顺序推进：

1. 先保证现有主流程稳定，不因为新需求破坏学习链路
2. 新增功能前先补接口、数据库和 Redis 约定
3. 优先补测试、示例数据和文档，再考虑扩大功能边界
4. 只有在需求明确时，才新增新的页面或业务模块
